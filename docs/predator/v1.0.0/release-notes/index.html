Predator - Release Notes

Version 1.0.0​

Release Date: June 2025
Status: General Availability (GA)

We're excited to announce the first stable release of Predator — a scalable, high-performance model inference service built as a wrapper around NVIDIA Triton Inference Server, part of BharatMLStack. Predator serves Deep Learning and tree-based models with low latency in Kubernetes, integrates with the Online Feature Store (OnFS) and uses Interflow for orchestration; clients send inference requests via the Helix client over gRPC.

What's New​

Triton Inference Engine​

  • NVIDIA Triton: Unified inference runtime for DL and tree-based models on CPU/GPU
  • Model repository: Local repo materialized via Init Container from cloud storage (e.g. GCS); no runtime dependency on remote storage
  • gRPC API: Efficient request/response handling; clients use the Helix client to talk to Triton inside the Predator pod

Multi-Backend Support​

Pluggable backends for a variety of model types from a single deployment:

BackendUse Case
TensorRTGPU-optimized DL; serialized engines (FP16/INT8)
PyTorchNative PyTorch via LibTorch
ONNX RuntimeFramework-agnostic ONNX with TensorRT/GPU
TensorFlowSavedModel format
PythonCustom preprocessing, postprocessing, or unsupported models
FILTree-based models (XGBoost, LightGBM, Random Forest) on GPU
DALIGPU-accelerated data preprocessing
CustomC++/Python backends for proprietary runtimes

Dynamic Batching & Concurrency​

  • Dynamic batching: Multiple requests combined into a single batch at runtime — higher GPU utilization, improved throughput, configurable via config.pbtxt
  • Concurrent model execution: Multiple models or multiple instances of the same model; load distribution via instance_group
  • Model versioning & ensembles: Multiple versions per model; ensemble pipelines to reduce latency

Kubernetes Deployment​

  • Helm-based: Standardized, GPU-optimized model serving on K8s
  • Pod layout: Init Container (model sync from GCS) + Triton Inference Server Container
  • Custom Triton images: Production uses custom-built images (only required backends) for smaller size and faster startup; built on GCP VM, pushed to Artifact Registry
  • Image distribution: Secondary boot disk caching and optional image streaming for faster scale-up
  • Health probes: Readiness/liveness via /v2/health/ready; traffic only after models are loaded
  • Autoscaling: CPU-based and GPU-based (DCGM metrics) scaling

Observability​

  • Prometheus metrics: Latency, throughput, GPU utilization from the Triton Inference Container
  • Grafana: Visualization of inference and GPU metrics
  • Warmup requests: Configurable to preload kernels and avoid cold-start latency

APIs & Integration​

gRPC via Helix Client​

Client applications (e.g. Realestate, IOP) send inference requests through the Helix client, which communicates with the Triton Inference Server inside the Predator pod. No direct Inferflow/Predict RPCs on this service — inference is Triton-native with Helix as the client interface.

Ecosystem​

Deployment & Operations​

Container & Images​

  • Custom Triton images: Built on GCP VM, pushed to Artifact Registry; only required backends included
  • Optional response caching: Custom cache plugins at image build time
  • Init Container: Downloads model artifacts from GCS to populate the model repository before Triton starts

Supported Environments​

  • Kubernetes (K8s)
  • Google Kubernetes Engine (GKE) with GPU node pools
  • Amazon EKS (GPU-supported)

Compatibility​

Triton & Backends​

  • NVIDIA Triton Inference Server: Version aligned with custom image build
  • Backends: TensorRT, PyTorch, ONNX Runtime, TensorFlow, Python, FIL, DALI, Custom — select at image build time

Download & Installation​

Source Code​

git clone https://github.com/Meesho/BharatMLStack.git
cd BharatMLStack/predator

Deployment​

Predator is typically deployed via Helm onto a Kubernetes cluster. Custom Triton images are built, pushed to Artifact Registry, and referenced in the Helm chart. Model artifacts are stored in GCS; the Init Container syncs them to the pod's model repository before Triton starts.
